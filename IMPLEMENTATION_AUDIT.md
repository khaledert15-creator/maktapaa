# تدقيق تنفيذ مكتبة دوت كوم

تاريخ التدقيق: 16 يوليو 2026

## التقنية والبنية الحالية

- Monorepo باستخدام pnpm workspaces وTypeScript 5.9 وNode.js 24.
- واجهة React 19 وVite وTailwind CSS 4 وRadix UI وTanStack Query وWouter.
- API مبني بـ Express 5، مع جلسات مخزنة في PostgreSQL.
- PostgreSQL مع Drizzle ORM وDrizzle Kit، وعقد OpenAPI يولّد عميل React وZod schemas عبر Orval.
- لا توجد خدمة دفع مفعلة؛ الدفع عند الاستلام هو المسار الوحيد المسموح به من الخادم والواجهة.

## مكتمل ويعمل

- واجهة عربية RTL متجاوبة: الرئيسية، الكتالوج، البحث، تفاصيل المنتج، السلة، إتمام الطلب، تأكيد الطلب، التسجيل والدخول والحساب.
- API للمنتجات والتصنيفات والمحتوى والشحن والسلة والحسابات والطلبات.
- لوحة إدارة عربية: الملخص، قائمة المنتجات، إضافة وتعديل وأرشفة المنتجات، قائمة الطلبات وتفاصيل الطلب وتغيير حالته.
- مخطط منتجات وتصنيفات ومخزون وعملاء وطلبات وكوبونات ومحتوى ومراجعات وجلسات.
- طلب إلغاء منفصل يقدمه العميل ويوافق أو يرفض عليه موظف؛ العميل لا يغيّر حالة الطلب مباشرة.
- تتبع تاريخ حالة الطلب وحركات خصم واستعادة المخزون.
- استعادة جلسة العميل أو الموظف بعد تحديث الصفحة، وحماية صفحات الإدارة.
- صلاحيات خادم لمسارات المنتجات والطلبات مع تجاوز آمن للمالك ومدير النظام.
- فرض الدفع عند الاستلام داخل API ورفض أي قيمة Fawry حاليًا.
- أول migration كاملة محفوظة في `lib/db/drizzle`.
- typecheck والبناء الإنتاجي ينجحان.

## مكتمل جزئيًا

- الكوبونات موجودة، لكن شروط تاريخ الصلاحية والحد الأدنى والحد الأقصى وعدد الاستخدامات تحتاج تطبيقًا موحدًا داخل transaction.
- المفضلة والعناوين لها عقود API ومخطط قاعدة بيانات، لكن بعض endpoints ما زالت ترجع بيانات مؤقتة أو فارغة.
- تقارير ولوحات الملخص لها API أولي، لكن لا توجد كل شاشات الإدارة المقابلة.
- Fawry موجود كقيمة مستقبلية في enum والعقد، لكنه معطل في الواجهة ومرفوض من API؛ لا يوجد تكامل مزعوم.
- التكامل المستقبلي مع نظام المكتبة الداخلي غير منفذ، ولا توجد ادعاءات بأنه مكتمل.

## مفقود

- شاشات إدارة العملاء والمخزون والكوبونات والشحن والتصنيفات والمحتوى والتقارير والموظفين والصلاحيات.
- فاتورة/إيصال غير ضريبي قابل للطباعة باسم «مكتبة دوت كوم».
- واجهة العميل لإرسال طلب الإلغاء ومتابعة قراره بصورة واضحة داخل تفاصيل الطلب.
- سجل تدقيق عام لكل تعديل حساس للموظفين؛ الموجود حاليًا يغطي حالة الطلب وحركة المخزون.
- اختبارات آلية للوحدات والتكامل وE2E، ولا توجد أوامر lint/test في المستودع.
- رفع صور آمن؛ حقل الغلاف حاليًا يعتمد على رابط خارجي.
- استعادة كلمة المرور رغم وجود رابط في صفحة الدخول.

## أعطال تم إصلاحها في هذا العمل

- فشل TypeScript بسبب `.tsbuildinfo` قديمة وعدم إجبار بناء مكتبات composite.
- خطأان في session helpers كانا يستخدمان متغيرًا غير موجود.
- استدعاءات React Query المولدة كانت غير متوافقة مع النوع الحالي وكانت تمنع البناء.
- ثلاث صفحات إدارة حرجة كانت TODO: نموذج المنتج وقائمة الطلبات وتفاصيل الطلب.
- جلسة المصادقة كانت تضيع من حالة React بعد تحديث الصفحة.
- CORS كان يقبل أي origin فعليًا.
- إعداد الجلسة كان يسمح بسر افتراضي وcookie غير آمنة في الإنتاج.
- API كان يقبل Fawry إذا أرسله العميل يدويًا.
- تكرار طلب الإلغاء والقرارات غير الصحيحة لم يكونا ممنوعين.
- خصم واستعادة مخزون الطلب لم يكونا يسجلان حركة مخزون.
- لم تكن هناك migrations محفوظة.

## مخاطر أمنية متبقية

- لا يوجد rate limiting لمحاولات الدخول أو endpoints العامة.
- لا توجد حماية CSRF صريحة للطلبات المعتمدة على cookies؛ SameSite=Lax يقلل الخطر لكنه ليس بديلًا كاملًا.
- المراجعات الجديدة تُعتمد تلقائيًا، وتحتاج moderation وصلاحية شراء مؤكدة.
- إدخالات عدة endpoints تُفحص يدويًا بدل Zod schemas موحدة، ما يزيد احتمال mass assignment أو قيم غير صالحة.
- لا توجد سياسة قوية لرفع/عرض الصور أو Content Security Policy.
- بيانات الدخول الافتراضية في seed يجب ألا تستخدم في الإنتاج، ويجب تدويرها عند أول تشغيل.

## مشكلات قاعدة البيانات المتبقية

- إنشاء الطلب وتحديث الكوبون والمخزون يجب نقله بالكامل إلى transaction ذرية لمنع حالة جزئية أو overselling عند التزامن.
- يلزم indexes على حقول البحث والتصفية والعلاقات كثيرة الاستخدام مثل status، created_at، customer_id، product_id.
- حالة طلب الإلغاء مخزنة كنص بدل enum، ولا يوجد unique partial index يمنع أكثر من طلب pending لكل طلب.
- علاقات الموظف في assigned/status history/movements ليست foreign keys في عدة مواضع.
- migration مولدة لكن لم تُطبق لعدم توفر `DATABASE_URL` في بيئة العمل الحالية.

## مشكلات UX/UI متبقية

- bundle الواجهة أكبر من 1MB ويحتاج route-level code splitting.
- بعض روابط لوحة الإدارة تقود إلى صفحة غير موجودة حتى تُنفذ الوحدات المقابلة.
- إجراءات قبول/رفض الإلغاء تظهر في تفاصيل كل طلب؛ الأفضل إظهارها فقط عند وجود طلب pending مع السبب.
- رسائل أخطاء API في بعض النماذج عامة ولا تعرض سبب التحقق العربي القادم من الخادم.
- رحلة checkout تحتاج اختبار تكامل فعلي بقاعدة بيانات، وحالات empty/loading/error أكثر تفصيلًا.

## نتائج التحقق

- `pnpm run typecheck`: ناجح.
- `pnpm run build`: ناجح للواجهة وAPI وmockup sandbox.
- `pnpm --filter @workspace/db run generate`: ناجح وأنشأ migration أولية.
- lint/tests: لا توجد scripts أو suites حالية؛ مسجل كعمل مفقود وليس نجاحًا مزعومًا.
- تشغيل الواجهة: ناجح على المنفذ 5173 واستجابة HTTP 200.
- تشغيل API ورحلات قاعدة البيانات: متوقف على توفير PostgreSQL و`DATABASE_URL` و`SESSION_SECRET` للإنتاج.
